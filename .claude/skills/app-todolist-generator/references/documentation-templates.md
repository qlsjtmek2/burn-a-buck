# Documentation Templates

투두리스트를 문서화할 때 사용할 수 있는 마크다운 템플릿 모음입니다.

## Development Plan Template

프로젝트 개발 계획서 템플릿입니다.

```markdown
# [프로젝트명] - 개발 계획

## 📋 프로젝트 개요

- **생성일**: YYYY-MM-DD HH:MM:SS
- **작성자**: Claude Code (app-todolist-generator)
- **프로젝트 타입**: [앱 타입]
- **복잡도**: [간단/중간/복잡] (예상 기간)
- **총 작업 수**: XX개
- **예상 기간**: XX일

## 🛠️ 기술 스택

- **플랫폼**: mobile/web
- **프레임워크**: React Native/React
- **백엔드**: Supabase/Firebase
- **상태관리**: Zustand + React Query

## ✨ 주요 기능

- 기능 1
- 기능 2
- 기능 3

## 🎯 Phase 1: 프로젝트 초기 설정

- [ ] 프로젝트 초기화 및 기본 구조 설정
- [ ] 필수 패키지 설치
- [ ] 프로젝트 폴더 구조 구성

## 🎯 Phase 2: 백엔드 설정

- [ ] 백엔드 프로젝트 생성 및 설정
- [ ] 데이터베이스 스키마 설계
- [ ] API 클라이언트 설정

## 🎯 Phase 3: 핵심 기능 구현

- [ ] 메인 화면 UI 구현
- [ ] CRUD 기능 구현
- [ ] 추가 기능 구현

## 🎯 Phase 4: UI/UX 개선

- [ ] 로딩 상태 UI 구현
- [ ] 에러 처리 UI 구현
- [ ] 반응형 디자인 적용

## 🎯 Phase 5: 테스트 및 배포

- [ ] 기능 테스트 및 버그 수정

- **완료**: 0/XX (0.0%)
- **진행 중**: Phase 1
- **예상 남은 시간**: XX일

## 📝 메모

- 중요한 결정 사항이나 이슈 기록

---

_이 문서는 Claude Code의 app-todolist-generator 스킬로 자동 생성되었습니다._
```

## Architecture Decision Record (ADR) Template

중요한 아키텍처 결정을 문서화하는 템플릿입니다.

```markdown
# ADR-XXX: [결정 제목]

## 상태

- **제안일**: YYYY-MM-DD
- **결정일**: YYYY-MM-DD
- **상태**: 제안/승인/거부/폐기

## 맥락

어떤 상황에서 이 결정이 필요했는지 설명합니다.

- 현재 문제점
- 제약 조건
- 요구사항

## 결정

무엇을 결정했는지 명확하게 기술합니다.

우리는 [선택한 방안]을 사용하기로 결정했습니다.

## 고려한 대안

### 대안 1: [대안명]

**장점:**
- 장점 1
- 장점 2

**단점:**
- 단점 1
- 단점 2

### 대안 2: [대안명]

**장점:**
- 장점 1
- 장점 2

**단점:**
- 단점 1
- 단점 2

## 결정 근거

왜 이 방안을 선택했는지 상세히 설명합니다.

1. 이유 1
2. 이유 2
3. 이유 3

## 결과

이 결정으로 인한 영향과 결과를 기술합니다.

**긍정적 영향:**
- 영향 1
- 영향 2

**부정적 영향 (Trade-offs):**
- 영향 1
- 영향 2

**주의사항:**
- 주의할 점 1
- 주의할 점 2

## 관련 결정

- ADR-XXX: [관련 결정]
- ADR-XXX: [관련 결정]

---

_작성자: Claude Code_
```

## Retrospective Template

Phase 완료 후 회고 템플릿입니다.

```markdown
# [프로젝트명] - Phase X 회고

## 📅 회고 정보

- **Phase**: Phase X - [Phase명]
- **기간**: YYYY-MM-DD ~ YYYY-MM-DD
- **참여자**: 팀원 이름들
- **작성일**: YYYY-MM-DD

## ✨ 잘한 점 (Keep)

1. **[항목]**
   - 구체적인 설명
   - 왜 잘했는지
   - 다음에도 계속할 방법

2. **[항목]**
   - 설명

## 🔧 개선할 점 (Problem)

1. **[항목]**
   - 무엇이 문제였는지
   - 왜 문제가 되었는지
   - 영향도

2. **[항목]**
   - 설명

## 💡 시도할 점 (Try)

1. **[항목]**
   - 구체적인 액션 아이템
   - 담당자
   - 예상 효과
   - 적용 시기

2. **[항목]**
   - 설명

## 📊 통계 및 메트릭

- **계획 대비 실제**
  - 계획: X일
  - 실제: Y일
  - 차이: +/-Z일

- **작업 완료율**
  - 완료: X개
  - 미완료: Y개
  - 추가: Z개

- **예상 vs 실제 시간**
  - Task 1: 예상 Xh / 실제 Yh
  - Task 2: 예상 Xh / 실제 Yh

## 🎓 학습 내용

### 기술적 학습

- 새로 배운 기술이나 패턴
- 유용했던 라이브러리나 도구
- 적용 가능한 베스트 프랙티스

### 프로세스 개선

- 효과적이었던 작업 방식
- 개선이 필요한 프로세스
- 도입할 만한 도구나 방법론

## 🎯 다음 Phase 목표

1. **목표 1**
   - 구체적인 내용
   - 달성 기준

2. **목표 2**
   - 구체적인 내용
   - 달성 기준

## 📝 추가 메모

- 기타 특이사항
- 공유할 내용
- 참고 자료

---

_작성일: YYYY-MM-DD_
```

## Usage Guide

### 문서 생성 워크플로우

1. **개발 계획 단계**
   - Development Plan Template 사용
   - 초기 투두리스트 생성
   - 기술 스택 및 기능 정의

2. **개발 진행 단계**
   - Quick Task Checklist로 세부 작업 관리
   - ADR로 중요 결정 기록
   - 정기적으로 Progress Report 작성

3. **Phase 완료 단계**
   - Retrospective Template으로 회고 진행
   - 학습 내용 정리
   - 다음 Phase 계획 수립

4. **프로젝트 완료 단계**
   - Project Summary Template으로 최종 정리
   - 전체 통계 및 성과 정리
   - 향후 개선 계획 수립

### 문서 저장 구조

```
claudedocs/
├── development-plan.md          # 메인 개발 계획
├── progress-reports/             # 진행 보고서
│   ├── week-1.md
│   ├── week-2.md
│   └── phase-1.md
├── decisions/                    # 아키텍처 결정
│   ├── ADR-001-tech-stack.md
│   ├── ADR-002-state-management.md
│   └── ADR-003-api-design.md
├── retrospectives/               # 회고
│   ├── phase-1-retro.md
│   ├── phase-2-retro.md
│   └── phase-3-retro.md
├── checklists/                   # 작업 체크리스트
│   ├── feature-a.md
│   └── feature-b.md
└── project-summary.md            # 최종 요약

```

### 문서 작성 베스트 프랙티스

1. **일관성 유지**
   - 모든 문서에 동일한 템플릿 사용
   - 날짜 형식 통일 (YYYY-MM-DD)
   - 상태 표시 통일 ([ ], [~], [x], [!], [?])

2. **정기적 업데이트**
   - Development Plan: 작업 완료 시 즉시 업데이트
   - Progress Report: 주간 또는 Phase 단위로 작성
   - Retrospective: 각 Phase 완료 직후 작성

3. **구체적 작성**
   - 추상적인 표현보다는 구체적인 수치와 사실 기록
   - "잘했다"보다는 "X 기능을 Y 방식으로 구현하여 Z 효과를 얻었다"
   - 학습 내용은 재사용 가능하도록 상세히 기록

4. **버전 관리**
   - 모든 문서를 Git으로 관리
   - 코드 변경과 함께 문서 커밋
   - 주요 마일스톤마다 태그 생성

5. **팀 공유**
   - 문서는 팀 전체가 접근 가능한 위치에 저장
   - 중요한 결정사항은 ADR로 남기기
